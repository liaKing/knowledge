RAG 技术介绍

## 1. 什么是 RAG

*   **定义**：检索增强生成（Retrieval-Augmented Generation）
*   **核心概念**：结合信息检索与生成式AI的混合架构
*   **发展历程**：从传统搜索到智能问答的演进
*   **应用价值**：提高回答准确性，减少幻觉，增强可控性，提供可追溯的信息来源

简单来说：让AI模型能够访问和利用外部知识库

## 3. RAG 详细介绍

### 3.1 阶段介绍

一般情况下 RAG 分为几种阶段：朴素 RAG（Naive RAG），进阶 RAG（Advanced RAG），模块化RAG（Modular RAG）。正常情况下 Naive RAG 仅仅只是实现了 RAG 最基本的功能，往往是一个产品为快速实现做的 demo 所用到的，但是也是初学者快速学习和理解RAG原理的教学工具；Advanced RAG 则是当前企业级应用的主流选择（事实上很多企业连这个级别也没有达到但功能依然够用）他所能提供的业务支持包括但不限于：智能客服，知识库问答，Saas 产品的 RAG 模块；Modular RAG 代表RAG技术的最新发展方向，复杂场景的终极解决方案和大型科技公司的技术竞争优势级别的，适用场景：复杂多模态应用，大规模企业级系统，前沿研究项目。当然 RAG 作为最近几年新出的技术其技术规范并不固定，这里只是其中一种理论介绍。

### 3.2 流程介绍

**Naive RAG 流程介绍**：分为两大步即：1. 向向量数据库插入文本向量，2. 从向量数据库中检索相似度匹配最高的数据，但是具体方案里面细节的流程差距很大，这里做最简单的介绍

### 3.3 RAG 关键步骤介绍

#### 3.3.1 切片

RAG中的切片（Chunking）虽然看似基础，但实际上具有举足轻重的地位，它是整个RAG系统的基石组件，RAG 的切片处理往往是整个方案效果的关键。他的作用有多重要呢？举个例子：假设一位老人写了几十年多日记去世了，当他过世之后他的儿子让一位编辑为他整理一篇老人打篮球的成长经历，编辑接到任务之后让编辑的助手从几十年的日记中找到一些片段记录在一些笔记上供他参考，在这个流程中助手选取的片段就是切片，如果助手选取的片段选择的语义不全或者有歧义都会影响编辑的写作。

##### 3.3.1.1 主流切片的方式：

1.  **固定长度切片**
    *   `chunk_size = 512`
    *   `tokens overlap = 50` (重叠tokens)
    *   **优点**：
        *   ✅ 实现简单，计算效率高
        *   ✅ 内存占用可预测
        *   ✅ 适合批量处理
    *   **缺点**：
        *   ❌ 容易切断语义单元
        *   ❌ 忽略文档结构
        *   ❌ 可能破坏上下文连贯性
    *   基本不推荐，最简单最垃圾的一种方式

2.  **递归切片（常见的切片方式）**
    *   递归切片通过分隔符优先级和递归分解的方式，实现更加自然和语义连贯的文本切分，按分隔符优先级递归分割，直到满足大小要求
    *   **Step1**: 尝试用高优先级分隔符分割
    *   **Step2**: 如果片段仍过大，用次级分隔符继续分割
    *   **Step3**: 重复直到所有片段符合要求

3.  **Claude 的缓存生成切片（BM256+向量）**
    *   原始文档：`https://www.anthropic.com/news/contextual-retrieval`
    *   **主要方案思路**：
        *   将整个文档给到大模型
        *   将langchan分好片的切片给大模型，大模型根据之前缓存的文档为切片提供其所在的上下文信息，拼成新的切片（大模型缓存可以节省大量成本也是此方案能使用的主要原因）
        *   使用BM25对切片中的关键字进行BM25评分，评分越高代表此关键字与这个文档的相关性越高（考虑了此关键字在整个文档的频率和此切片的频率）
    *   **用户查询**：
        *   获取用户的问题的关键字，和向量
        *   分布根据关键字和向量获取最相关度切片
        *   重排序

    *   **BM25得分**：
        *   假设有一个词"apple"：
            *   在当前文档中出现了3次（TF = 3）
            *   总文档数是1000（N = 1000）
            *   包含"apple"的文档有10个（df = 10）
        *   则：
            *   IDF = log(1000/10) = log(100) ≈ 4.605
            *   TF-IDF = 3 × 4.605 = 13.815

    *   **哪些向量数据库支持BM25**
        *   **OpenSearch**
            *   支持方式: 原生支持
            *   版本要求: 2.x及以上
            *   特点:
                *   基于Elasticsearch
                *   支持k-NN和BM25混合搜索
                *   提供可调节的相关性参数
            *   示例代码:
                ```json
                {
                  "query": {
                    "hybrid": {
                      "queries": [
                        {
                          "match": {
                            "content": {
                              "query": "search text",
                              "boost": 0.3
                            }
                          }
                        },
                        {
                          "knn": {
                            "vector_field": {
                              "vector": [...],
                              "k": 10,
                              "boost": 0.7
                            }
                          }
                        }
                      ]
                    }
                  }
                }
                ```
        *   **Weaviate**：原生支持
        *   **Elasticsearch**：原生支持

    *   **总结**：切片方式有非常多的方式，但具体都是要结合方案来的，不过虽然切片对于效果的作用非常重要，但是大多数应用选择的都是递归切片，主要原因就是简单，快速

#### 3.4.2 LightRAG
    *   GitHub开源项目，内有详细教程和运行步骤
        *   `https://github.com/HKUDS/LightRAG`
    *   LightRAG是一个轻量级的检索增强生成框架，其主要特点是：
        *   **核心特征**：
            *   双层检索系统
            *   轻量级设计
            *   高效性能
        *   **主要创新**：
            *   低层级知识检索
            *   高层级知识检索
            *   信息综合处理
        *   **优势**：
            *   速度更快
            *   资源消耗低
            *   实现简单
        *   **应用场景**：
            *   对性能要求高的环境
            *   资源受限的系统
            *   需要快速响应的应用
    *   它的代码已在GitHub上开源，可以直接使用和参考。
    *   相比传统RAG，LightRAG的主要改进在于它的轻量化设计和双层检索机制，使得整个检索增强生成过程更加高效。

    *   **LightRAG 流程介绍**：分块——》并发提取实体和关系——〉合并——》向量化——〉存储

    *   **问题**：提取的实体和关系去重之后进行向量化了没有？
        *   **答案**：进行向量化了，例如：
            ```json
            entities = [
                {
                    'name': 'OpenAI',
                    'type': '科技公司',
                    'description': '全球领先的人工智能研究机构，致力于开发安全和有益的人工智能',
                    'key_attributes': ['AI研究', 'GPT模型', '语言模型'],
                    'related_relations': ['Sam Altman 是 CEO', '微软是主要投资者']
                },
                {
                    'name': 'Sam Altman',
                    'type': '企业家',
                    'description': 'OpenAI 现任 CEO，硅谷知名科技创新者',
                    'key_attributes': ['AI创业', '风险投资', '技术前瞻'],
                    'related_relations': ['OpenAI 创始人之一', 'Y Combinator 前总裁']
                }
            ]
            ```
            上面是两个实体，对上面的数据进行向量化。
            实体可以理解为关系型数据库中的数据，关系就是关系表，理论上任何两个实体都有关系，但是这里只会保存有意义的关系。
            如果A和B,C,D都是朋友，那么会有三条关系数据且都是朋友关系。
            如果A和B又是领导关系，那么A和B之间的关系数据会有两条，分别是朋友关系和领导关系。

    *   **问**：如何对文本块提取实体和关系呢？
        *   **答**：通过 NER 模型进行提取，gpt4o-mini 也可以。
    *   **问**：提取的实体和关系对后面的查询有什么帮助吗？
        *   **答**：为实体数据提供结构化的表示，且可以通过实体之间的关系找到相关的信息，可以通过提取的实体和关系构建知识图谱。

### 3.6 RAG beyond Transformer

这段文字介绍了除基于Transformer模型之外，还存在使用检索增强（Retrieval-Augmented Generation, RAG）技术的大模型架构。这些架构可能基于不同的深度学习技术，例如LSTM（长短时记忆网络）、Diffusion模型和GAN（生成对抗网络），并应用于各种生成任务。

以下是具体例子和相关应用领域：

1.  **Audio（音频生成）**：
    *   **示例研究**：检索增强的语音生成模型（RETRIEVAL-AUGMENTED TEXT-TO-AUDIO GENERATION）。
    *   **描述**：通过使用检索增强技术，改进了文本到音频的生成。这意味着在生成音频时，模型能够利用相关的检索信息来产生更自然或更符合人类期望的声音输出。
2.  **Video（视频生成）**：
    *   **示例研究**：Animate-A-Story: Storytelling with Retrieval-Augmented Video Generation。
    *   **描述**：在视频生成领域，通过检索增强技术来支持内容生成，例如通过检索与故事内容相关的视频片段来增强故事讲述的效果。这使得生成的视频更具相关性和连续性。
3.  **Code（代码生成等）**：
    *   **描述**：虽然具体细节未列出，但也有模型在代码生成或其他类似任务中使用检索增强技术。这种方法可用于提高代码的准确性、效率和可读性。

**总结**：RAG技术并不限于单一的Transformer架构。其他深度学习模型，例如LSTM、Diffusion模型和GANs，都可以通过结合检索增强来提升生成任务的效果。在音频生成、视频生成，乃至代码生成等多种领域应用这些技术，可以更好地利用上下文关联信息以提升生成质量。

## 5. 向量模型选择

向量模型（或称嵌入模型）是RAG系统中的核心组件之一，它们负责将文本（或其他数据类型）转换成高维度的数值向量，这些向量能够捕捉文本的语义信息。选择合适的向量模型对于RAG系统的检索质量和整体性能至关重要。

**选择向量模型时需要考虑的关键因素**：

*   **性能与准确性**：模型生成向量的质量直接影响检索结果的相关性。需要评估模型在特定任务（如语义相似度、问答）上的表现。
*   **效率与速度**：模型的推理速度和资源消耗（内存、计算）会影响RAG系统的实时性和可扩展性。
*   **语言支持**：如果应用需要处理多语言内容，则需要选择支持相应语言的向量模型。
*   **模型大小与部署**：大型模型通常性能更优，但部署成本和延迟也更高。需要根据实际部署环境和需求进行权衡。
*   **领域适应性**：通用型模型适用于广泛的场景，但针对特定领域（如医疗、法律）可能存在更专业的模型，能提供更精准的嵌入。
*   **成本**：商业API模型通常按使用量收费，而开源模型则需要考虑自托管的计算资源成本。

**主流向量模型介绍**：

*   **BERT (Bidirectional Encoder Representations from Transformers)**:
    *   **特点**：由Google开发，基于Transformer架构，通过预训练学习语言的深层双向表示。其派生模型（如Sentence-BERT, MiniLM）广泛用于生成句子和段落级别的嵌入。
    *   **优势**：在多种NLP任务上表现出色，语义理解能力强。
    *   **应用**：文本相似度、信息检索、问答系统。
*   **RoBERTa (A Robustly Optimized BERT Pretraining Approach)**:
    *   **特点**：Facebook AI改进的BERT版本，通过更大的数据集、更长的训练时间和动态掩码策略，进一步提升了性能。
    *   **优势**：通常比BERT表现更好，尤其是在下游任务上。
*   **GPT系列 (Generative Pre-trained Transformer)**:
    *   **特点**：OpenAI开发，虽然主要以生成能力闻名，但其内部的Transformer编码器部分也能用于生成高质量的文本嵌入。例如，`text-embedding-ada-002`是OpenAI提供的一个高效的嵌入模型。
    *   **优势**：强大的通用语义理解能力，尤其适合复杂和开放域的文本。
*   **Word2Vec / GloVe**:
    *   **特点**：较早期的词嵌入模型，将每个词映射到一个向量。通过对词向量进行平均或加权平均可以得到句子或文档的粗略向量表示。
    *   **优势**：计算效率高，适用于资源受限的场景。
    *   **局限性**：无法捕捉词语在不同上下文中的多义性，句子/文档嵌入质量相对较低。
*   **FastText**:
    *   **特点**：Facebook AI开发，在Word2Vec基础上引入了子词信息，能够处理未登录词（OOV）问题，并支持多语言。
    *   **优势**：对OOV词有较好的处理能力，训练速度快。
*   **BGE (BAAI General Embedding)**:
    *   **特点**：由北京智源人工智能研究院（BAAI）开发，专注于生成高质量的通用文本嵌入，在中文和英文等多种语言上表现优异。

*   **E5系列 (Embeddings from Encoder-Decoder models)**:
    *   **特点**：微软开发，基于编码器-解码器架构，通过对比学习进行训练，旨在生成高质量的通用文本嵌入。

    *   **应用**：高级语义搜索、跨模态检索、复杂问答系统。



## 6. 其他技术

### 6.1 PQ 量化

PQ量化（Product Quantization）是向量数据库中一种重要的向量压缩和近似搜索技术，能够显著减少存储空间和加速相似度搜索。详细内容请参见 [7.2.4 PQ 量化](#724-pq-量化)。

### 6.2 BQ 量化 (Binary Quantization)

BQ量化（Binary Quantization）是一种将浮点向量转换为二进制向量（即每个维度只有0或1）的压缩技术。它通过将向量的每个维度映射到单个比特位来实现极致的压缩。

*   **定义**：BQ量化是一种将浮点向量转换为二进制向量（即每个维度只有0或1）的压缩技术。它通过将向量的每个维度映射到单个比特位来实现极致的压缩。
*   **原理**：
    1.  **阈值化**：通常通过一个阈值（如0或向量维度的均值/中位数）来决定每个维度是编码为0还是1。
    2.  **哈希函数**：也可以通过学习一个哈希函数，将相似的浮点向量映射到相似的二进制码。
*   **优点**：
    *   **极致压缩**：存储空间可以压缩到原始浮点向量的1/32或1/64。
    *   **极速搜索**：相似度计算可以转换为汉明距离（Hamming distance）计算，速度非常快。
*   **缺点**：
    *   **严重精度损失**：相比PQ量化，BQ量化通常会带来更大的精度损失，因为每个维度只保留了1比特信息。
    *   **搜索准确性较低**：在某些场景下，由于精度损失过大，搜索结果的相关性可能不理想。
*   **应用场景**：对存储空间和搜索速度有极高要求，且对精度损失容忍度较高的场景，例如大规模图像检索的初步筛选阶段。

### 6.3 Ragas

Ragas (Retrieval Augmented Generation Assessment) 是一个用于评估RAG（检索增强生成）系统性能的框架。随着RAG技术在各种应用中的普及，如何客观、有效地评估其生成质量和检索效果变得至关重要。Ragas 提供了一套全面的指标和工具，帮助开发者和研究人员量化RAG系统的各个方面。

*   **Ragas 是什么？**
    *   Ragas 是一个开源的评估框架，专注于衡量RAG系统在生成答案的**准确性 (faithfulness)**、**相关性 (relevance)**、**一致性 (consistency)** 和**上下文召回 (context recall)** 等方面的表现。它旨在自动化RAG系统评估过程，减少人工评估的成本和主观性。
*   **作用**：
    *   **量化评估RAG性能**：提供一系列指标，帮助用户了解RAG系统在不同维度上的表现，例如生成的答案是否忠实于检索到的上下文，检索到的上下文是否与问题相关，以及答案是否连贯一致。
    *   **指导RAG系统优化**：通过对不同组件（如检索器、生成器、分块策略等）的评估，Ragas 可以帮助识别系统中的瓶颈，从而指导优化方向。
    *   **促进迭代开发**：使开发者能够快速测试和比较不同RAG配置的效果，加速RAG应用的开发和改进周期。
    *   **减少幻觉**：通过评估答案的忠实性，Ragas 有助于发现和减少RAG系统生成“幻觉”（即模型生成的不真实或无根据的信息）的倾向。
*   **原理**：
    *   Ragas 的评估原理主要基于对RAG系统输出的多个维度进行量化分析。它通常需要以下输入：
        *   **问题 (Question)**：用户向RAG系统提出的查询。
        *   **上下文 (Context)**：RAG系统从知识库中检索到的相关信息。
        *   **答案 (Answer)**：RAG系统基于上下文生成的问题答案。
        *   **参考答案 (Ground Truth Answer)** (可选)：用于更精确评估答案准确性的真实答案。
    *   基于这些输入，Ragas 计算一系列指标：
        *   **忠实性 (Faithfulness)**：衡量生成的答案在多大程度上可以从检索到的上下文中推断出来。高忠实性意味着答案没有引入新的、未经证实的信息。
        *   **相关性 (Answer Relevance)**：评估生成的答案与原始问题之间的相关程度。
        *   **上下文相关性 (Context Relevance)**：评估检索到的上下文与原始问题之间的相关程度。不相关的上下文会稀释有用信息，甚至引入噪声。
        *   **上下文召回率 (Context Recall)**：衡量检索到的上下文是否包含了回答问题所需的所有关键信息。
        *   **上下文精确率 (Context Precision)**：衡量检索到的上下文中有多少是真正有用的信息。
    *   这些指标通常通过结合LLM（大型语言模型）的判断能力和传统的NLP技术来计算。LLM可以被用来判断答案的逻辑一致性、与上下文的匹配度等，而传统的相似度算法则用于衡量文本之间的语义距离。

#### 6.3.1 Ragas 运行流程

Ragas 的运行流程通常遵循以下步骤：

1.  **数据准备**：
    *   收集或生成用于评估的数据集，包括：
        *   `questions` (问题列表)
        *   `contexts` (RAG系统检索到的上下文列表)
        *   `answers` (RAG系统生成的答案列表)
        *   `ground_truths` (可选，人工标注的参考答案列表)
    *   将这些数据组织成Ragas可接受的格式，通常是Pandas DataFrame或类似结构。

2.  **选择评估指标**：
    *   根据评估目标，选择Ragas提供的相关指标，例如 `faithfulness` (忠实性)、`answer_relevance` (答案相关性)、`context_relevance` (上下文相关性)、`context_recall` (上下文召回率) 等。

3.  **配置LLM**：
    *   Ragas 内部使用LLM来计算某些指标（例如，判断答案是否忠实于上下文）。需要配置用于评估的LLM模型，可以是OpenAI、Anthropic或其他兼容的LLM提供商。

4.  **执行评估**：
    *   调用Ragas的评估函数，传入准备好的数据、选择的指标和LLM配置。
    *   Ragas 会自动对每个问题-上下文-答案对计算所选指标的分数。

5.  **结果分析**：
    *   Ragas 会返回一个包含所有指标分数的报告，通常包括平均分、标准差等统计信息。
    *   分析这些结果，识别RAG系统的优点和缺点，例如：
        *   如果 `faithfulness` 分数低，可能意味着生成模型存在幻觉。
        *   如果 `context_relevance` 分数低，可能意味着检索器不够准确。
        *   如果 `context_recall` 分数低，可能意味着检索器未能召回所有必要信息。

6.  **迭代优化**：
    *   根据评估结果，对RAG系统的组件（如分块策略、嵌入模型、检索算法、提示工程等）进行调整和优化。
    *   重复评估流程，直到达到满意的性能。

最后还有很多提升 RAG 效果的技术，例如意图识别和质量检查，问题拆分，多路召回，重排序，方式方法多种多样，具体方案还是要具体考虑。

## 7. 向量搜索算法

在RAG系统中，高效地从海量向量中找到与查询向量最相似的向量是核心挑战。这被称为**最近邻问题 (Nearest Neighbor Problem)**。

**最近邻问题**：给定一个查询向量，从一个大型向量集合中找到与其“最近”的K个向量。这里的“最近”通常通过距离度量（如欧氏距离）或相似度度量（如余弦相似度）来定义。

**相似度度量**：
1.  **余弦相似度**：两个向量的夹角越小越相似，常用于文本语义相似度。
2.  **欧氏距离**：两个向量在多维空间中的直线距离越近越相似，常用于图像特征匹配。

为了解决最近邻问题，主要有两种方法：

### 7.1 暴力搜索 (Brute-Force Search / Flat Search)

*   **原理**：计算查询向量与数据库中所有向量的距离或相似度，然后排序找出最近的K个。
*   **优点**：
    *   **准确率100%**：保证找到真正的最近邻。
    *   **实现简单**。
*   **缺点**：
    *   **计算成本高**：随着向量数量和维度的增加，计算量呈线性增长，在大规模数据集中不可行。
    *   **查询延迟高**。
*   **应用场景**：数据集规模小，或对准确率有绝对要求且计算资源充足的场景。

### 7.2 近似最近邻算法 (Approximate Nearest Neighbor, ANN)

为了克服暴力搜索的效率问题，ANN算法牺牲了一定的准确率，以换取更快的搜索速度和更低的资源消耗。

#### 7.2.1 K-means 聚类算法

K-means聚类算法是一种简单且常用的无监督机器学习算法，广泛应用于数据分析和模式识别领域。它的目标是将给定的数据集划分成多个簇（clusters），从而最小化簇内数据点之间的距离平方和。

*   **K-means算法的基本步骤**：
    1.  **选择簇的数量 (K)**：首先需要选择要划分的簇数K。这通常是根据先验知识或领域经验来决定的。
    2.  **初始化簇中心（质心）**：随机选择K个数据点作为初始的簇中心（centroids），或者使用其他启发式方法进行初始化（比如K-means++）。
    3.  **分配数据点到最近的簇中心**：对于每个数据点，计算其与所有簇中心的距离（通常是欧氏距离），将该数据点分配到距离最近的簇中心对应的簇中。
    4.  **更新簇中心**：对于每个簇，重新计算簇中心，即将簇内所有数据点的平均值作为新的簇中心。
    5.  **重复步骤 3 和 4**：重复数据点分配和簇中心更新的步骤，直到簇中心不再发生变化（即收敛），或者达到预设的迭代次数限制。
*   **特点和优缺点**：
    *   **优点**：
        *   简单易懂，易于实现。
        *   计算速度快，对大数据集也能较快收敛。
        *   在簇形状是圆形时效果较好。
    *   **缺点**：
        *   需要事先指定簇的数量K。
        *   对于不同的初始值，可能收敛到不同的局部最优解。
        *   对噪声和异常点较为敏感。
        *   适用于线性可分的、球形的簇，不适合处理复杂形状的簇。
*   **改进和变体**：
    *   **K-means++**：通过一种特定的初始中心选择策略来提高算法的收敛速度和精度。
    *   **Mini-batch K-means**：在每次迭代中使用随机选取的小批量数据来更新簇中心，提高了处理大规模数据集时的效率。
    *   **Hierarchical K-means**：结合层次聚类的思想进行分层聚类。
*   **应用**：K-means广泛应用于数据挖掘、图像处理、客户细分、市场调查、模式识别等领域，帮助发现数据中的自然聚集现象或模式。

#### 7.2.2 LSH (Locality Sensitive Hashing) 位置敏感哈希函数

*   **原理**：LSH通过设计特殊的哈希函数族，使得相似的向量以较高的概率映射到相同的哈希桶中，而不相似的向量则以较低的概率映射到相同的桶中。在搜索时，只需检查查询向量所在的哈希桶中的少量向量，从而大大减少搜索范围。
*   **优点**：
    *   **搜索速度快**：尤其适用于高维数据。
    *   **内存效率高**：存储哈希值通常比存储原始向量更节省空间。
*   **缺点**：
    *   **精度损失**：哈希碰撞可能导致相似向量被分到不同桶，或不相似向量被分到相同桶，从而影响召回率和精确率。
    *   **参数调优复杂**：哈希函数的选择和参数设置对性能影响很大。
*   **应用场景**：超大规模数据集的近似搜索，如大规模图像/视频检索、去重、推荐系统等。

#### 7.2.3 算法选择的4要点与常规建议

**考虑的4要点**：
1.  **查询延迟 (Query Latency)**：用户等待搜索结果的时间。
2.  **准确率 (Recall/Accuracy)**：找到相关结果的比例。
3.  **可用内存资源 (Memory Footprint)**：索引和存储向量所需的内存量。
4.  **索引延迟 (Indexing Latency)**：将新向量添加到索引所需的时间。

**常规建议**：
1.  **如果内存不是瓶颈**：优先考虑HNSW (Hierarchical Navigable Small World) 算法，它在查询延迟和准确率方面都能得到很好的保障。
2.  **如果希望内存使用更少，且索引速度比HNSW快，同时还能保持类似的查询质量**：则可以考虑IVF (Inverted File Index) 算法。
3.  **如果内存是较大瓶颈**：可以考虑给HNSW或IVF加入PQ编码（Product Quantization），但需要注意准确率可能会下降，查询延迟可能上升。

**向量化过程中速度、质量、内存都很重要。**

#### 7.2.4 PQ 量化 (Product Quantization)

PQ量化（Product Quantization）是向量数据库中一种重要的向量压缩和近似搜索技术，能够显著减少存储空间和加速相似度搜索。

*   **定义**：PQ量化是一种将高维向量分解为多个低维子向量，并对每个子向量独立进行量化的技术。通过这种方式，可以大幅度压缩向量的存储空间，并加速近似最近邻（ANN）搜索。
*   **原理**：
    1.  **向量分解**：将原始D维向量分解为M个D/M维的子向量。
    2.  **子向量聚类**：对每个子向量集合独立进行K-means聚类，生成K个码本（codebook）。每个码本包含K个聚类中心。
    3.  **编码**：每个子向量被其最近的聚类中心索引替代。因此，一个D维向量被编码为M个整数（每个整数代表一个子向量的码本索引）。
*   **优点**：
    *   **显著压缩存储空间**：存储M个整数远小于存储原始浮点向量。
    *   **加速搜索**：相似度计算可以在压缩后的空间进行，或者通过查表（lookup table）的方式加速。
    *   **可扩展性**：适用于大规模数据集。
*   **缺点**：
    *   **精度损失**：量化过程是近似的，会引入一定的精度损失，可能影响搜索结果的准确性。
    *   **码本生成开销**：K-means聚类生成码本需要一定的计算资源和时间。
*   **应用场景**：大规模向量检索系统，如图像检索、推荐系统、语义搜索等，尤其是在内存或带宽受限的环境中。
